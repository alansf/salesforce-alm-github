### Concrete facts the additional docs add to the plan

**Sandbox tier reference table** (Platform Guide pg 04) — for inclusion in Pattern 8 as a quick-reference:

| Sandbox Type | Fee to Salesforce | Storage Size | Refresh Interval | Type of Data |
|---|---|---|---|---|
| Developer | Free | 200 MB | 1 day permissible | Metadata only |
| Developer Pro | 5% of net spend | 1 GB | 1 day permissible | Metadata only |
| Partial Copy | 20% of net spend | 5 GB | 5 days permissible | Metadata + fraction of production data |
| Full Copy | 30% of net spend | Same as production org | 28 days permissible | Metadata + all production data |

**The 5 data seeding challenges** (Platform Guide pg 06) — frame Pattern 8 around these because they're the customer-facing pain articulation:

1. Maintaining parent-child relationships (Sandbox refreshes won't pick up related records unless objects are tied via "master-detail" or "required lookup" — random samples in Partial Copy break referential integrity)
2. Filtering and refining test data and attachments (Data Loader requires .CSV manipulation; polymorphic fields, intra-object relationships, and attachment storage limits compound)
3. Seeding on-demand (development cycles outpace refresh capability; teams hand-tune CSVs only to need to redo it when requirements shift)
4. Protecting confidential data (sandbox data contains PII; GDPR/CCPA/HIPAA/PCI-DSS impose strict controls; third-party contractors amplify risk)
5. Keeping data consistent across orgs (without comparison tooling, the only diff option is V-Lookup across multiple .CSV files in Excel)

**The 4 typical approaches** (Platform Guide pg 08–09) — Pattern 8 recommends approach 4:

1. **Sandbox Refresh** — refresh from prod; Developer/Pro only get metadata; refresh frequency is rate-limited
2. **Sandbox Cloning** — clone an existing sandbox once you have an ideal data set; preserves data + metadata
3. **Data Loader** — bulk .CSV import/export; "be prepared to set aside a couple of extra weeks to fully replicate the necessary objects and their dependencies"
4. **Salesforce Data Mask & Seed** (Recommended) — customize sandbox data, seed specific objects/records, filter field values, add new or update existing

**NIST citation** (Platform Guide pg 03): "Companies spend an average of 30x more to fix bugs released into production." — anchor for ROI section of Pattern 8.

- **Salesforce DevOps Center** — free, admin-accessible, narrowing the Flosum moat for non-regulated shops

### 1.2 `feedback_llm_runtime_placement.md` (type: feedback)

The architectural rule about where LLMs belong in ALM tooling.

**Rule**: LLMs belong at *build time* (developer authoring code, generating workflows, scaffolding) — never in the *runtime data path* of a deployment or seeding pipeline.

**Why**: runtime LLM use is non-deterministic, slow (5–30 sec per inference), token-expensive at scale, and unauditable. Auditors can't accept "the AI picked these records."

**How to apply**: when a customer or TA proposes "AI-driven seeding" or "AI-driven deployment," push the LLM use back to build time. The runtime should be deterministic Apex / YAML / SQL / Bash.

**Exception**: LLM-assisted *curation* (a one-off "pick representative records for this test scenario" call) is fine because the output is a deterministic config artifact that downstream code consumes.

### 1.3 `feedback_data_mask_seed_native_first.md` (type: feedback)

The specific Salesforce-ALM rule that emerged from the docs.

**Rule**: when a customer is doing sandbox seeding and/or data masking, the default recommendation is **Salesforce-native Data Mask & Seed**, not third-party tools.

**Why** (anchored to the docs):
- It's on-platform — within the Salesforce Trust Boundary, no external SaaS in the data path
- FedRAMP, HIPAA, SOX certified out of the box
- 3.5M records/hr masking throughput beats most third-party tools' batch performance
- $0 Seed add-on when sold with Data Mask, so cost is bounded (10% Net AOV for Data Mask + $0 for Seed)
- Bypasses all automations including managed-package automations during seeding
- Starter templates for CPQ, FSL, nCino are bundled
- Schedule seeds + masks to run automatically post-refresh
- Customer evidence: AGCO (2–4 weeks → <24 hrs), MSU (1 day → <1 hr), Barracuda (replaced Prodly), Make-A-Wish (40 hrs saved per refresh, 90% faster)

**When to deviate** (defensible non-default):
- Customer has heavy CPQ/Industries (SBQQ, OmniStudio, Vlocity) data deployment that pre-dates Native and has already invested in Prodly's data model — switching cost may exceed value
- Customer is on Accelerate today and renewal is imminent — migration to Data Mask & Seed is the recommendation, not staying on Accelerate
- Customer has Flosum and the bundled Data Migrator covers their CPQ flavor adequately — use what's paid for
- Data Cloud / Data 360 seeding is "Coming Soon" per the FY27 roadmap — for DC-heavy customers today, the native answer isn't fully there yet

### 2.1 New Pattern 8: Sandbox lifecycle with Data Mask & Seed

Add to `~/.claude/skills/salesforce-alm-github/SKILL.md` between Pattern 7 (Unlocked Packages) and the existing "Additional Key Workflows" section.

**Pattern 8 structure** (mirrors how Patterns 1–7 are written):

- **The Problem**: sandboxes start either empty (Developer) or with a random subset (Partial Copy capped at 10K records/object) or all of production with PII (Full Copy). None of those is a usable test environment. Manual seeding takes 1–4 weeks per refresh; manual masking is error-prone and slow
- **The Solution**: chain the native sandbox lifecycle — *refresh → mask → seed* — using Salesforce Data Mask & Seed on Core, scheduled and automated. No third-party tool required for the common case
- **Workshop Value**: "AGCO went from 2–4 week sandbox refresh cycles to under 24 hours. MSU went from 1 day to <1 hour. Barracuda replaced Prodly with this and reduced refresh times from 2 weeks to 4 days. The native tooling is now faster than the third-party alternatives that preceded it."
- **The four-phase model** (anchored in docs):
  1. **Refresh** — Salesforce performs sandbox refresh (Partial or Full)
  2. **Mask** — Data Mask runs (per template); de-identifies PII at the database layer using one of nine masking strategies: Replace with Library, Pattern Masking, Anonymize, Custom Value, Fill if Empty, Existing Value Random, Partial List, Deletion, Partial Value
  3. **Seed** — Data Seed populates curated test records; supports synthetic data, master-detail awareness, starter templates (CPQ, FSL, nCino), and bypass of automations including managed-package automations
  4. **Available** — sandbox is now compliant + test-ready; users can begin work
- **Implementation steps**:
  - Configure Data Mask templates per sObject classification (PII, Confidential, Internal)
  - Define seeding policies (filter records, set hierarchical relationships, enable schedule)
  - Wire the chain via Data Mask & Seed UI scheduling: "Run after refresh or on a schedule"
  - For CI/CD orchestration: GitHub Actions cron triggers `sf org refresh`, post-copy hook fires Mask + Seed automatically based on template association with the sandbox
- **Common gotchas** (from the doc + inference):
  - Tenant model: Data Mask runs on force.com (the Salesforce platform); Seed runs on the Own infrastructure (managed by customer post-acquisition). Customers need to manage both. Single Seed tenant can serve multiple orgs — don't create multiple Seed tenants
  - Data Mask Managed Package end-of-service: **Nov 2026** — customers on the legacy package must migrate before then
  - Own Accelerate end-of-life: **Dec 2027** — existing Accelerate customers should renew onto Data Mask + $0 Seed SKU at renewal time, not extend Accelerate
  - GovCloud / FedRAMP customers: Salesforce Data Seeding Gov is also $0 add-on to Data Mask (FedRAMP Moderate)
  - Data Cloud / Data 360 seeding (DLO → DLO) is *Coming Soon* on the FY27 roadmap, not GA today
- **When to use**: any Salesforce shop running ≥1 sandbox refresh per quarter, especially regulated industries (HIPAA, SOX, FedRAMP) or shops with admin-heavy teams who need a native UI rather than custom scripts
- **When the v2.1.0 pattern is not enough**: heavy SBQQ CPQ + OmniStudio data movement at scale where Prodly's pre-built Apps still beat Data Seed's templates; Data 360 / Data Cloud seeding (until that's GA); customers locked into Flosum Data Migrator on a current contract
